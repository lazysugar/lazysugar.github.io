SpringMVC是如何注册Controller的？

Spring是通过RequestMappingHandlerMapping的Bean来注册Controller的。

简单打个springboot的项目来追踪下：

来到RequestMappingHandlerMapping的抽象父类AbstractHandlerMethodMapping实现了InitializingBean接口。对所有 Bean初始化时候会执行afterPropertiesSet -> getHandlerInternal方法，可以看到我们的indexController

调用processCandidateBean，其中会调用isHandler方法，判断该Bean是否是web资源类——是否被@Controller或@RequestMapping注解或他的派生注解标记


判断成功后就会调用detectHandlerMethods进行解析Method。上部分代码就是对Bean对象的所有Method进行解析，selectMethods中第二个参数是一个回调方法getMappingForMethod里面进行解析。

通过createRequestMappingInfo方法把method和handlerType作为为参数，创建了RequestMappingInfo对象，将两个对象结合并返回。

然后来到detectHandlerMethods方法的下半部分，将刚刚获取到的RequestMappingInfo对象集合，调用registerHandlerMethod遍历注册。

调用mappingRegistry的register方法来注册

然后我们来看看DispatcherServlet是如何处理请求信息的，直接看到doService -> doDispatch

doDispatch会调用getHandler,根据Mapping信息调用其getHandler方法，里面存在着我们的RequestMappingHandlerMapping。

最终调用到lookupHandlerMethod，会根据我们的请求路径lookupPath,来通过mappingRegistry寻找对应的mapping，就是RequestMappingInfo对象。

所以如果要实现内存马的话就需要创建一个RequestMappingInfo对象然后注册到IOC容器中。

Springboot 2.6.0 修改

在 Springboot 2.6.0 环境下注入Controller内存马后会报500错误，错误提示 java.lang.IllegalArgumentException: Expected lookupPath in request attribute “org.springframework.web.util.UrlPathHelper.PATH”

原因在于从 Springboot 2.6.0 版本开始，官方修改了url路径的默认匹配策略，版本发布日志部分如下：

追踪下看看：
在调用RequestMappingInfo#getMatchingCondition的时候，判断patternsCondition不为空时调用getMatchingCondition方法

最终调用到getResolvedLookupPath，里面会断言lookupPath为空，导致抛出异常

所以用PatternsRequestCondition创建访问 controller 的 URL 地址就不能用了

PatternsRequestCondition url = new PatternsRequestCondition("/1azys");

如果在 Springboot 2.6.0 的环境下，通过 application.properties配置文件设置spring.mvc.pathmatch.matching-strategy的值为ant_path_matcher，即修改服务端的路径匹配策略为 AntPathMatcher，注入的Controller内存马后访问就没问题了。

我们看看springboot是怎么创建RequestMappingInfo对象的，来到RequestMappingHandlerMapping#createRequestMappingInfo

参照这段代码来得到RequestMappingInfo对象

RequestMappingInfo info = RequestMappingInfo.paths("/lazy")
        .options(config)
        .build();
